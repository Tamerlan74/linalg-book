# Суть

В разборе задачи — **отдельный шаг проверки ответа**, разными средствами. Это и педагогика (учим проверять), и страховка от ошибки.

Виды проверок, которые применяются:

| Проверка | Когда применима |
|---|---|
| Крайние случаи | Если ответ зависит от параметра — подставить 0, 1, бесконечность. |
| Симметрия | Если задача симметрична — ответ тоже симметричен. |
| Размерность | В физических задачах — проверка единиц. |
| Знак | Площадь не отрицательная, вероятность не больше 1. |
| Альтернативный путь | Решить другим способом, сравнить. |
| Связанные тождества | Проверить через известное тождество (например, $\det(BA) = \det B \det A$). |

# Пример из главы 4

> **Sanity check.** ... Произведение определителей: $\det A \cdot \det B = -6$.
> 
> Теперь посчитаем $\det(BA)$ напрямую. ... Совпало с $\det A \cdot \det B = -6$. Это и есть та самая проверка: если бы мы ошиблись где-то в умножении матриц или в счёте определителя — числа не совпали бы.

# Когда применять

- В **3-5 задачах** из блока на главу. Не во всех — иначе паттерн обесценивается.
- Для **задач со счётом**, где есть риск арифметической ошибки.
- В **смешанных задачах**, где синтез нескольких приёмов даёт ловушки.

# Когда не применять

- В **тривиальных задачах** — sanity check будет искусственным.
- В **доказательствах** — там сам метод доказательства = проверка.

# Структура

```
Задача N. <условие>.

**Решение.** <решение>. <Ответ: X.>

**Sanity check.** <способ проверки>. <Проверка показывает X. Совпало.>
```

# Главное правило

Sanity check **не повторяет** основное решение. Он использует **другие свойства**, **другие тождества**, **другой подход**. Иначе это не проверка, а копия.

# Связка

Очень сильная связка: `tasks_multiple_solutions` + `tasks_sanity_check`. Решили двумя способами, совпало — это и есть встроенная проверка. Для самых важных задач — рекомендуется.
