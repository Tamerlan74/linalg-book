# Суть

Для **избранных задач** (1-2 на главу) показываются **2-3 разных способа решить**. Зачем: читатель учится не одному способу решать одну задачу, а **множеству стратегий мышления**. Узнаёт, что **стратегия — это инструмент**, а не «правильный способ».

# Пример из главы 4

В разделе про умножение матриц 3×3 одна и та же задача решена двумя способами:
1. **Строка на столбец** (стандартно) — быстрее на бумаге.
2. **Через столбцы** — глубже по смыслу.

Оба способа дают одинаковый результат. Читатель видит, что **выбор способа — это навык**.

# Когда применять

- Для **показательных** задач, где разные подходы дают **разный объём работы**.
- Для **классических** задач, где альтернативный способ красивее и поучительнее стандартного.
- НЕ для каждой задачи — иначе блок задач превращается в учебник.

# Структура

```
Задача N. <условие>.

**Способ 1 (стандартный).** <решение>. <Результат: X.>

**Способ 2 (альтернативный).** <решение>. <Результат: X.> 

Видите разницу? Способ 1 — <характеристика>. Способ 2 — <характеристика>. 
В разных задачах удобнее разное.
```

# Когда альтернативный способ обязателен

- Когда у задачи есть **симметрия**, которую видно одним способом и не видно другим.
- Когда **прямой метод** даёт длинный счёт, а **умный** — короткий.
- Когда **алгебра** и **геометрия** работают одинаково (паттерн пересекается с `derive_geom_alongside`).

# Связка

`tasks_multiple_solutions` хорошо сочетается с `tasks_sanity_check`: если решение даётся двумя способами, и оба дают одно — это **естественная** проверка. Совпадение = взаимная страховка.

# Сборщик

Этот паттерн **не для каждой задачи**. На этапе 5 пайплайна:
- Скрипт-генератор задач выбирает 1-2 задачи как «кандидатов на multiple solutions».
- Для этих задач **отдельный промпт LLM**: «реши эту задачу двумя способами, объясни разницу».
- SymPy проверяет, что оба решения дают один результат (защита от расхождения).
