# Суть

Когда автор **сознательно упрощает** материал (опускает строгое доказательство, обходит технический нюанс), он **прямо это говорит** — не делает вид, что строгости нет, и показывает, куда смотреть, если читатель захочет точнее. Это сохраняет доверие.

Thompson в главе II про степени малости делает аккуратную сноску о различии «order of magnitude» и «smallness» — мимоходом, не отвлекая. Книга упрощает, **но читатель знает, что упрощение есть**.

# Шаблон

> Строго говоря, здесь надо было бы <что точно>. Для нашей задачи это не критично; если интересно — см. <источник или будущая глава>.

# Когда применять

- При **опускании строгого доказательства** (например, доказательства через перестановки).
- При **обходе технических ограничений** (например, «формула работает для $a, b, c, d > 0$, общий случай — аналогично»).
- При **сокращении** ради лёгкости чтения.

# Когда не применять

- Если **не упрощено ничего** — не выдумывать упрощения для красоты.
- Если упрощение **существенно меняет смысл** — тогда это не упрощение, а другой материал; нужно либо доказывать строго, либо переписывать.

# Конфликт

`derive_honest_skip` **конфликтует** с `derive_numerical_check` в **одном блоке выкладки**. Если выкладку упростили, численная проверка по этой же выкладке нечестна — числа покроют упрощение. Если упрощение есть, численная проверка должна быть на **отдельном примере** с явным указанием «вот пример, где упрощение работает».

См. `00_conflicts.md`.

# Хорошие места для применения

| Тип упрощения | Шаблон фразы |
|---|---|
| Опущено доказательство | «Подробное доказательство есть в любом учебнике, нам важно...» |
| Обойдены ограничения | «Эта формула верна для общего случая; мы показали идею на положительных числах.» |
| Упрощена форма | «Строгое определение через перестановки — позже, в главе X. Пока работает разложение по строке.» |
| Опущена общность | «В размерности 3 это так. Для большей размерности — рекурсивно по тому же принципу.» |
