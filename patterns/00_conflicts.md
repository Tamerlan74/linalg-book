# Конфликты и совместимости паттернов

Этот файл — централизованная таблица **парных отношений** между паттернами. Сборщик читает его при формировании промпта для конкретной главы. Если в плане главы указаны два конфликтующих паттерна — сборщик выдаёт ошибку и не собирает.

Отношения бывают трёх типов:

- **CONFLICT** — паттерны нельзя применять одновременно в одной единице текста (точная единица указана). Это жёсткое ограничение.
- **REDUNDANCY** — паттерны делают похожее, оба сразу — это переигрывание. Допустимо, но требует обоснования.
- **SYNERGY** — паттерны усиливают друг друга, рекомендованная связка.

## CONFLICTS — жёсткие запреты

| Паттерн 1 | Паттерн 2 | Уровень конфликта | Объяснение |
|---|---|---|---|
| `intro_disarm_symbols` | `(прямая подача формальной нотации)` | глава | Целевой читатель — «с нуля». Нотацию нельзя подавать без разоружения. Если в каком-то разделе формальная нотация даётся прямо — нужно отметить как сознательное отступление. |
| `open_self_deprecation` | `open_manifesto` | абзац | Самоуничижение Thompson и манифест-самоутверждение Savov создают противоречивый голос в одном абзаце. В одной главе — допустимо, если в разных разделах. |
| `open_self_deprecation` | `open_great_minds_erred` | раздел | Оба снимают страх ошибки разными средствами. Подряд — переигрывание. Выбрать один. |
| `derive_honest_skip` | `derive_numerical_check` | один блок выкладки | Если выкладку упростили («строго говоря, тут надо так, но...»), численная проверка по этой же выкладке нечестна — числа покроют упрощение. Если упрощение есть, численная проверка должна быть на отдельном примере с явным указанием. |

## REDUNDANCY — допустимо, но требует обоснования

| Паттерн 1 | Паттерн 2 | Уровень | Объяснение |
|---|---|---|---|
| `intro_analogy_first` | `intro_etymology` | абзац | Оба делают «лёгкий вход» в понятие. Можно вместе, если аналогия из жизни, а этимология даёт второй смысловой якорь. Не использовать как замену друг друга. |
| `pause_historical_anecdote` | `pause_real_world_use` | раздел | Обе — передышки. Подряд в одном разделе — слишком много отступлений. Чередовать по главам. |
| `tasks_multiple_solutions` | `tasks_sanity_check` | задача | Оба «обогащают» задачу. Если задача и так показывает 3 способа решения, sanity check добавляет четвёртый шаг — может перегрузить. Использовать вместе только для самых важных задач. |
| `climax_name_the_aha` | `climax_card_summary` | раздел | Оба маркируют кульминацию. Если назван «момент главы» И показана карточка-сводка — это уже двойная подсветка. Допустимо, но не превращать в правило. |

## SYNERGY — рекомендованные связки

Это пары/тройки паттернов, которые работают **особенно сильно вместе**. Когда сборщик видит один паттерн из связки в плане главы — он может подсказать включить второй для усиления.

### Связка «Мягкий вход в главу»

`open_self_deprecation` + `open_naming_enemy` + `open_story_first`

«Я тоже не понимал» → «учебники подают это как наказание» → «вот сценка из жизни». Это **стандартная заставка главы** для нашей книги, проверенная на главах 1-4.

### Связка «Введение нового понятия с нуля»

`intro_analogy_first` + `intro_named_model` + `intro_etymology`

Бытовая аналогия → даём модели имя → объясняем, откуда взялось слово. Три якоря: аналогия (интуиция), имя (мнемоника), этимология (смысл).

### Связка «Вывод формулы через смысл»

`derive_geom_alongside` + `derive_case_structure` + `derive_numerical_check`

Геометрия + 3 case-а + численная проверка. Это последовательность из главы 3 (вывод $ad-bc$). Когда срабатывает — формула выглядит не магической, а очевидной.

### Связка «Кульминация»

`climax_name_the_aha` + `climax_card_summary` + `derive_geom_alongside` (как опора)

Назвать момент + показать визуальную карточку + опираться на уже введённую геометрию. Это та самая «карточка четырёх ответов» из главы 3.

### Связка «Биохазард с возвратом»

`biohazard_marker` + `derive_anticipate` + `tasks_sanity_check`

Биохазард сразу после изложения нового правила (через предвосхищение «вы можете подумать, что...») + потом в задачах sanity check, который проверяет именно эту ошибку. Это связка из главы 4 — порядок умножения матриц.

### Связка «Закрытие главы»

`close_what_we_learned` + `close_bridge_promise`

Сводка + обещание. Стандартное завершение каждой главы.

## Как сборщик использует этот файл

При сборке промпта для конкретной фазы главы:

1. Читает список паттернов, заданных в плане главы для этой фазы.
2. Проверяет таблицу CONFLICTS — нет ли запрещённых пар. Если есть — **отказ собирать**, лог ошибки.
3. Проверяет таблицу REDUNDANCY — нет ли «переигрываний». Если есть — **предупреждение** в логе, но сборка идёт.
4. Проверяет таблицу SYNERGY — может ли предложить добавить парный паттерн. Если может — **подсказка** в логе.

Это даёт три уровня контроля: жёсткий запрет, предупреждение, подсказка.

## Эволюция этого файла

По мере написания книги обнаружатся новые конфликты и новые связки, о которых сейчас не догадываемся. Файл пополняется **после каждой главы**, на которой обнаружено что-то новое. Это и есть один из артефактов «реальное применение покажет», о котором говорилось при выборе пути B (компактные черновики + развитие).
